# 无按键语音唤醒方案设计

## 背景

当前 DJI Mic Mini 听写方案需要手动按按钮来开始和结束听写。虽然已经比键盘操作方便很多，但每次都要伸手去按胸口的按钮，仍然有物理操作的负担。

目标是实现完全不按按键的语音唤醒：把 DJI Mic Mini 夹在衣领上，通过一个自定义的触发声（口哨、唤醒词等）来控制听写的开始、结束和发送，全程不碰任何设备。

## 核心约束

- 触发方式必须稳定、可重复
- 误触率要极低（日常对话、环境噪音、音乐不能误触发）
- 声音小的时候也能准确识别
- 每个人可以自定义自己的触发方式（这是个人设备，不需要通用性）
- 触发声不一定是语音或文字，可以是任何能准确复刻的声音

## 务实版推荐方案

- 语音唤醒应直接驱动现有 `save/watch/preconfirm/confirm` 状态机，不另起一套发送逻辑；同一个唤醒词按状态解释为“开始 / 结束 / 发送”。
- 部署继续走现有 CLI：基础工作流仍由 `install/update/doctor/config` 管理；交互式 `install` 完成后应直接进入 cue 录制；唤醒词样本采集与校准走 `wakeword setup`、`wakeword record`、`wakeword train`、`wakeword doctor`，运行时监听走 `wakeword start/stop/status/listen`。
- 当前实现主线改成“个性化 vocal cue + 学习型 log-mel 分类器”，不再默认押注文字短词；推荐稳定、短促、不会自然出现在正文里的 cue，比如 `轻嘶两下`、`双击气音`、`click-hiss`。
- 训练端要内建少样本增广：增益扰动、轻微变速、时间平移、背景噪声混合和轻量 spec-mask 都应默认开启，用来补足个人样本量小的问题；推理端仍保持轻量。
- 训练建议走“个人定制”而不是“通用中文识别”：默认同时采主 cue 和 cancel cue 的 quiet / clean / noisy 正样本；speech decoy、其他口腔声音 decoy 和 ambient 负样本只录一套，供两类 cue 共用。
- 推荐状态机是：`idle + 命中 => start`，`recording + 命中 => stop/watch`，`ready-to-send + 3 秒内再次命中 => confirm/send`，超时则静默回到 `idle`。
- 推理建议用近讲麦 + 小模型流式检测 + 连续多帧确认 + 分状态阈值；开始可偏灵敏，结束更严格，发送最严格，并配一个短冷却避免连击。
- 换环境后不要全量重训；只补录 5 到 10 条“当前环境低声样本”和 30 到 60 秒背景噪声，做一次增量校准就够了。
- `doctor` 应主动发现“新麦克风 / 新办公室 / 连续误拒”并提示补录；用户也应该能手动触发一次快速补样。
- 录音交互不要做成长表单；用一个很小的浮窗录音器做 `Space 开始 / Space 结束`，根据系统语言切中文或英文，并支持英文长文案换行，再由 CLI 驱动短轮次采样。
- 每条录音控制在 0.8 到 1.5 秒，录完立刻回放并允许重录；这样比一次性录很多条更容易拿到稳定样本。
- 非语义声音完全可以训练，但前提是你能稳定复现；不要选自然咳嗽这种每天都在变的声音，优先选你能故意做出来的人工 cue。

## 方案一：口哨 Pattern 检测（可选）

### 原理

口哨是一种特殊的声音信号：频率集中在 1500-3500Hz 范围，几乎是纯正弦波，谐波极少。这使得它在频谱上表现为一个非常干净的尖峰，和任何语音、音乐都有本质区别。

检测口哨不需要任何机器学习，只需要基础的数字信号处理（FFT 频谱分析），20 行左右的代码就能实现。

### 为什么口哨比唤醒词更可靠

| 特性 | 口哨 | 唤醒词 |
|------|------|--------|
| 信号类型 | 纯音，确定性信号 | 语音，概率性识别 |
| 检测方法 | FFT + 阈值判断，逻辑确定 | 神经网络推理，存在置信度 |
| 误触率 | 接近零 | 0.1-1%（取决于词和环境） |
| 训练成本 | 零 | 需要录制样本或使用 SDK |
| 语言依赖 | 无 | 依赖语种支持 |
| 低音量表现 | 优秀（纯音尖峰容易检测） | 一般（音素特征会模糊） |

### 防误触：音乐和歌声的问题

纯频率检测确实不够。音乐里的笛子、长笛、甚至歌手哼唱都可能落在 1500-3500Hz。但可以通过音色检测来区分。

音色检测的原理是看频谱的「纯净度」：

| 声源 | 频谱特征 | 谐波数量 |
|------|---------|---------|
| 人的口哨 | 几乎纯正弦波，能量集中在基频，谐波极弱 | 1-2 个 |
| 笛子/长笛 | 有明显的 2、3、4 次谐波 | 4-8 个 |
| 人声哼唱 | 谐波丰富，频谱宽 | 10+ 个 |
| 音乐混音 | 频谱全频段都有能量 | 无法计数 |

检测方法是算 spectral purity（谱纯度）：基频能量占总能量的比例。口哨这个比例能到 0.8-0.95，笛子大概 0.3-0.5，混合音乐更低。加上这一个指标，音乐误触基本消除。

再叠加一层：DJI Mic Mini 是近讲麦克风，夹在领口距嘴约 15cm。使用者的口哨信号比远处播放的音乐响得多。设一个信噪比门槛，远处的音乐根本过不了。

最保险的做法是使用口哨 pattern——比如「短短长」（嘟嘟——嘟——），音乐里几乎不可能出现完全一样的时间节奏。相当于声音版的摩尔斯电码。

### 检测条件（多重叠加）

1. **频率范围**：基频在 1500-3500Hz 之间
2. **谱纯度**：基频能量 / 总能量 > 0.8
3. **持续时间**：持续 250ms 以上（过滤瞬态噪音）
4. **信噪比**：信号幅度显著高于背景噪声水平
5. **时间 pattern**（可选）：要求特定节奏（如两短一长）

这五个条件同时满足时才触发，误触率接近零。

## 方案二：自定义唤醒词

### 可选引擎

| 引擎 | 自定义词 | 中文支持 | 开源 | 上手难度 | 准确率 |
|------|---------|---------|------|---------|--------|
| Picovoice Porcupine | 输入短语即生成模型 | 支持 | 否（免费个人版） | 最低 | 高 |
| OpenWakeWord | 需要 ML 训练 | 仅英文 | 是（Apache 2.0） | 中 | 好 |
| microWakeWord | 需要手动调优 | 有限 | 是 | 中 | 高（设备端） |
| Sherpa-onnx | 支持 | 支持 | 是 | 中 | 好 |
| Mycroft Precise | 需要录音样本 | 有限 | 是 | 高 | 一般 |

如果用中文唤醒词，Picovoice Porcupine 是最省事的选择：输入短语就能生成模型，支持中文，免费个人使用。

### 唤醒词选择建议

「龙虾」太短了——两个音节，误触率会偏高。唤醒词引擎在嘈杂环境下容易把相似的音误判。建议 3-4 个音节：

- 「小龙虾」— 三音节，比「龙虾」稳定得多
- 「芝麻开门」— 四音节，经典，辨识度高
- 自造词也行，比如「吱吱嘎」— 越不像日常用语越好

用同一个词切换开始/结束完全没问题，就是 toggle 模式，和现在按钮逻辑一样。

### 防误触策略

1. **选择 3+ 音节的非常用词**：日常对话中不会出现的词
2. **置信度阈值设高**：设为 0.8+（默认通常是 0.5）
3. **声纹验证**：Picovoice 支持 Voice ID，录入几次声音后只响应注册者的声音
4. **音高二次验证**：检测到唤醒词后，验证说话者的基频是否在你的个人范围内
5. **冷却时间**：触发后 N 秒内忽略唤醒词，防止听写内容中出现唤醒词导致重复触发

## 低音量识别

### 硬件层面的天然优势

DJI Mic Mini 是近讲麦克风，这是最关键的优势。声音强度和距离的平方成反比。使用者的嘴距麦克风约 15cm，3 米外同事说话的声音到达麦克风时已经衰减了 (300/15)^2 = 400 倍。即使在低声耳语，信噪比依然很高。

加上 DJI Mic Mini 自带的两档主动降噪，环境噪音被进一步压制。

### 软件层面的增强

1. **AGC（自动增益控制）**：把音频信号归一化到固定响度再送检测。不管说多轻，检测引擎看到的信号幅度都一样。
2. **训练时包含低音量样本**：如果用唤醒词方案，录训练数据时刻意包含轻声说的版本，引擎就能学到低音量的特征。
3. **口哨方案更不用担心**：即使很轻的口哨，频谱上也是一个干净的尖峰，检测难度远低于语音。

## 实现路径

### 架构

```
DJI Mic Mini（领口发射器）
    → DJI 接收器（USB-C 连接 Mac）
        → macOS 音频输入设备
            → Swift listener 持续监听音频流
                → 切出候选片段后交给 Python detector 判分
                    → 命中时调用现有脚本状态机（save/watch/preconfirm/confirm）
                        → macOS 听写开始/结束/发送
```

不需要额外的音频路由软件（如 BlackHole、SoundSwitch）。当前落地方案是 `wakeword-listener.swift` 负责麦克风接入和切段，`wakeword_tools.py` 负责 learned log-mel 分类器打分和少样本增广训练。

### Phase 1：个性化 vocal cue + 全状态机（推荐先做）

零硬件成本，用现有的 DJI Mic Mini + Mac。

1. 录主 cue / cancel cue 的低声 / 自然 / 背景噪声正样本，再录一套共享的 speech decoy / mouth-sound decoy / ambient 负样本，生成本地 calibration.json
2. 用 `wakeword-listener.swift` 常驻监听 DJI Mic Mini 的音频输入
3. 候选片段交给 `wakeword_tools.py detect` 做 log-mel 分类判分
4. `toggle` 命中走 `dictation-enter.sh wakeword-toggle`，`cancel` 命中走 `dictation-enter.sh wakeword-cancel`
5. CLI 负责 `start/stop/status`，交互式 `install` 完成后直接进入 cue 录制，不另起第二套控制面

### Phase 2：口哨 / 非语义声音（只在短词不合适时考虑）

1. 复用同样的常驻监听架构
2. 把检测模块替换成 FFT + pattern 或自定义音频事件模型
3. 仍然沿用同一套状态机，不单独设计发送逻辑

### Phase 3：通用唤醒词引擎（未来可选）

1. 安装 Picovoice Porcupine 或 OpenWakeWord
2. 创建自定义中文唤醒词模型
3. 设置置信度阈值 0.8+
4. 按 `idle / recording / ready-to-send` 三个状态设置不同阈值
5. 保持同样的 listener + 状态机架构，只替换 detector backend 即可

### Phase 4：独立硬件（可选，脱离 Mac 近场限制）

只有在需要设备独立工作（不依赖 DJI 接收器连接 Mac）时才需要。

**方案 A：XIAO ESP32S3 Sense**
- 尺寸 21x17.5mm，内置 PDM 数字麦克风，$14
- 支持 BLE + WiFi
- 可在设备上运行 microWakeWord 或 Porcupine（推理 <10ms）
- 检测到唤醒词后通过 BLE 通知 Mac
- 3D 打印领夹外壳
- 300mAh 电池续航约 4-10 小时（始终监听模式）

**方案 B：Fork Omi（原名 Friend）开源项目**
- 基于 XIAO nRF52840 Sense，$16
- BLE 音频流传输到 Mac，Mac 端做检测
- 24 小时以上续航（100mAh 纽扣电池）
- 硬件设计、固件、外壳图纸全部开源
- GitHub: BasedHardware/omi

**DJI Mic Mini 固件改造不可行**：使用 Actions ATS2831 芯片，固件签名加密，无公开 SDK，无已知的自定义固件项目。

## 方案对比总结

| 方案 | 防误触 | 低音量表现 | 实现难度 | 硬件成本 |
|------|-------|-----------|---------|---------|
| 口哨 pattern + 谱纯度检测 | 极好 | 好 | 最低（纯 DSP，无 ML） | 零（用现有 DJI） |
| 中文唤醒词（Porcupine） | 好（3+ 音节） | 好（加 AGC） | 低（现成 SDK） | 零（用现有 DJI） |
| 口哨 + 唤醒词双因子 | 最好 | 一般 | 中等 | 零（用现有 DJI） |
| 独立硬件（ESP32S3） | 同上 | 同上 | 高 | ~$20 + 电池 + 外壳 |
